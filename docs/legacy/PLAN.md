# PLAN.md — План создания Telegram-бота для напоминания о приёме лекарств

## 1. Цель проекта

Цель проекта — создать Telegram-бота, который помогает пользователю не забывать принимать лекарства по заранее заданному расписанию.

Бот должен позволять пользователю добавлять лекарства, указывать дозировку, время приёма, периодичность напоминаний и фиксировать факт приёма. Основная задача бота — не давать медицинские рекомендации, а выступать удобной персональной напоминалкой.

Важно: бот не должен советовать дозировки, заменять врача, интерпретировать симптомы или самостоятельно менять схему лечения. Все данные о лекарствах, дозировках и расписании вводит сам пользователь.

---

## 2. Минимальная версия продукта — MVP

Первая рабочая версия бота должна включать только базовый, но полностью полезный функционал.

### MVP-функции

1. Регистрация пользователя после команды `/start`.
2. Добавление лекарства через пошаговый диалог.
3. Сохранение названия лекарства, дозировки и времени приёма.
4. Настройка периодичности приёма:
   - каждый день;
   - конкретные дни недели.
5. Отправка напоминания в нужное время.
6. Inline-кнопки под напоминанием:
   - `✅ Принял`;
   - `⏰ Напомнить через 10 минут`;
   - `⏭ Пропустить`.
7. Сохранение истории приёмов.
8. Просмотр списка добавленных лекарств.
9. Отключение или удаление лекарства.

## 3. Технический стек

Для проекта используется простой и практичный стек, подходящий для MVP и дальнейшего расширения.

### Основной стек

- Python
- aiogram 3
- SQLite
- SQLAlchemy
- APScheduler
- Docker / Docker Compose — опционально

### Почему SQLite

SQLite подходит для первой версии проекта, потому что:

- не требует отдельного сервера базы данных;
- легко запускается локально;
- хорошо подходит для небольших Telegram-ботов;
- упрощает разработку и тестирование;
- позволяет быстрее собрать MVP.

При росте проекта SQLite можно заменить на PostgreSQL без полной переработки архитектуры, если заранее использовать SQLAlchemy и не привязываться к специфичным возможностям SQLite.

## 4. Основные сущности базы данных

Для MVP достаточно четырёх основных сущностей.

### users

Хранит данные пользователей Telegram.

Поля:

- `id` — внутренний ID пользователя;
- `telegram_id` — Telegram ID пользователя;
- `timezone` — часовой пояс пользователя;
- `created_at` — дата регистрации.

### medicines

Хранит лекарства, добавленные пользователем.

Поля:

- `id` — ID лекарства;
- `user_id` — ID пользователя;
- `name` — название лекарства;
- `dosage_text` — дозировка в текстовом виде;
- `comment` — дополнительный комментарий;
- `is_active` — активно ли лекарство;
- `created_at` — дата добавления.

### medicine_schedules

Хранит расписание приёма лекарств.

Поля:

- `id` — ID расписания;
- `medicine_id` — ID лекарства;
- `time` — время приёма;
- `days_of_week` — дни недели;
- `snooze_minutes` — время отложенного напоминания;
- `remind_until_confirmed` — нужно ли повторять напоминания до подтверждения.

### intake_logs

Хранит историю приёма лекарств.

Поля:

- `id` — ID записи;
- `medicine_id` — ID лекарства;
- `scheduled_at` — на какое время был запланирован приём;
- `status` — статус: `taken`, `skipped`, `missed`;
- `responded_at` — когда пользователь нажал кнопку ответа.

## 5. Основные команды бота

### `/start`

Запускает бота, регистрирует пользователя и кратко объясняет назначение бота.

### `/add_medicine`

Запускает пошаговое добавление лекарства.

### `/my_medicines`

Показывает список активных лекарств пользователя.

### `/today`

Показывает лекарства, которые нужно принять сегодня.

### `/history`

Показывает историю приёмов.

### `/settings`

Открывает настройки пользователя:

- часовой пояс;
- настройки повторных напоминаний;
- стандартное время отложенного напоминания.

## 6. Пользовательский сценарий добавления лекарства

Пользователь вводит команду `/add_medicine`.

Бот последовательно задаёт вопросы:

1. Как называется лекарство?
2. Какая дозировка? Например: `1 таблетка`, `5 мл`, `2 капли`.
3. Во сколько напоминать?
4. Как часто принимать?
5. Нужно ли повторное напоминание, если пользователь не подтвердил приём?

После ввода данных бот показывает итоговую карточку:

```text
Лекарство: Магний
Дозировка: 1 таблетка
Время: 21:00
Частота: каждый день
```

Под карточкой выводятся кнопки:

- `Сохранить`;
- `Изменить`;
- `Отмена`.

После сохранения лекарство и его расписание записываются в базу данных.

---

## 7. Сценарий напоминания

В заданное время бот отправляет пользователю сообщение:

```text
Пора принять лекарство: Магний
Дозировка: 1 таблетка
```

Под сообщением отображаются кнопки:

- `✅ Принял`;
- `⏰ Напомнить через 10 минут`;
- `⏭ Пропустить`.

### Если пользователь нажал `Принял`

Бот записывает в историю статус `taken`.

### Если пользователь нажал `Пропустить`

Бот записывает в историю статус `skipped`.

### Если пользователь нажал `Напомнить через 10 минут`

Бот создаёт одноразовое отложенное напоминание.

### Если пользователь не ответил

Бот может автоматически записать статус `missed`, если включена соответствующая логика.

## 9. Этапы разработки

3. Установить зависимости.
4. Создать `.env` и `.env.example`.

6. Подключить aiogram.
7. Реализовать базовый обработчик команды `/start`.

Результат этапа: бот запускается и отвечает на `/start`.

---

### Этап 2. Подключение базы данных

Задачи:

1. Подключить SQLite.
2. Настроить SQLAlchemy.
3. Создать модели:
   - `User`;
   - `Medicine`;
   - `MedicineSchedule`;
   - `IntakeLog`.
4. Настроить создание таблиц.
5. Реализовать сохранение пользователя после `/start`.

Результат этапа: бот умеет сохранять пользователей в SQLite.

---

### Этап 3. Добавление лекарства

Задачи:

1. Создать FSM-состояния для добавления лекарства.
2. Реализовать пошаговый ввод:
   - название;
   - дозировка;
   - время;

   - настройки повтора.

4. Реализовать сохранение лекарства в базу.
5. Реализовать отмену добавления.

Результат этапа: пользователь может добавить лекарство и расписание.

---

### Этап 4. Просмотр лекарств

Задачи:

1. Реализовать команду `/my_medicines`.
2. Показать список активных лекарств пользователя.
3. Добавить кнопки управления лекарством:
   - посмотреть;
   - изменить;
   - отключить;
   - удалить.

Результат этапа: пользователь видит список своих лекарств.

---

### Этап 5. Планировщик напоминаний

Задачи:

1. Подключить APScheduler.
2. Настроить запуск планировщика вместе с ботом.
3. Загружать активные расписания из базы данных.

### Этап 6. Обработка действий пользователя

Задачи:

1. Реализовать inline-кнопку `Принял`.
2. Реализовать inline-кнопку `Пропустить`.
3. Реализовать inline-кнопку `Напомнить через 10 минут`.

Результат этапа: бот фиксирует реакцию пользователя на напоминание.

## 12. Важные ограничения

Бот должен соблюдать несколько принципов безопасности:

1. Не давать медицинских советов.
2. Не рекомендовать дозировки.
3. Не предлагать замену лекарств.
4. Не делать выводы о состоянии здоровья пользователя.
5. Не изменять расписание без явного действия пользователя.
6. Хранить только те данные, которые нужны для работы напоминаний.

---

## 13. Итоговая стратегия

Разработку стоит начинать с простой и надёжной напоминалки. Главная цель MVP — чтобы пользователь мог добавить лекарство, получить напоминание и отметить факт приёма.

После этого можно постепенно улучшать продукт: добавлять историю, статистику, расширенные настройки, курсы лечения и экспорт данных.

Главное — не усложнять архитектуру раньше времени. Для первой версии достаточно Python, aiogram, SQLite, SQLAlchemy и APScheduler.
